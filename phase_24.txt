3. User Ban — Atomic Transaction 
• Simulate: Admin bans user X who has 2 pending sessions and 1 locked escrow 
• Verify ALL of the following happen in SINGLE prisma.$transaction(): 
• User.status → BANNED 
• All PENDING sessions where user is mentor → CANCELLED 
• All PENDING sessions where user is student → CANCELLED 
• All LOCKED escrows for cancelled sessions → REFUNDED to respective students 
• Transaction records created for each refund 
• Verify: Cannot ban another ADMIN (role check returns 403) 
• Verify: Banned user cannot log in (auth check on login) 
• Verify: Banned user's active WebSocket connections are terminated 
4. Redis SCAN vs KEYS 
• CRITICAL: Search codebase for redis.keys('*') or client.keys('*') 
• Verify: All Redis key enumeration uses SCAN with cursor (not KEYS) 
• Reason: KEYS blocks the Redis event loop — in production with thousands of keys, this causes 
latency spikes and can freeze the entire Redis instance
