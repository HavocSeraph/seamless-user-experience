SECTION 1 — OVERALL WEBAPP FLOW 
1.1 Platform Philosophy 
SkillBarter is built on the Knowledge Time-Bank concept: education is the currency. Rather than 
charging money for learning, the platform creates a self-sustaining peer-to-peer economy where 
users teach to earn tokens, then spend tokens to learn. This directly addresses the core problem — 
financial barriers to upskilling — by replacing monetary transactions with a fair, gamified exchange 
mechanism. 
• Token Economy: SkillCoins (integer, non-divisible) are the sole medium of exchange 
• 50 coins on signup + 50 coins per OAuth verification (GitHub or LinkedIn) 
• Pricing tiers: Beginner (30 min) = 30 coins; Advanced (60 min) = 100 coins 
• Async-First Design: Bounty Board enables earning without scheduling live sessions 
• Post a question with a coin reward; experts answer with short video clips 
• Solves cold-start problem — new users can earn immediately 
• Escrow-Based Trust: tokens locked at booking, released only on mutual completion 
• Dispute mechanism freezes tokens pending admin review 
• No-show penalty: mentor cancels <1 hour before → loses 10 coins to student 
• 5% Transaction Tax: burned to admin wallet on every exchange to prevent inflation 
• Tax formula: Math.floor(amount × 0.05) — always floor division, never round 
1.2 The Golden Path User Journey 
1. SIGNUP: Email/password registration → bcrypt 12 rounds → 50 SkillCoins credited immediately 
via LedgerService.credit() → EARN transaction record created 
2. OAUTH VERIFICATION: Connect GitHub (pulls repos + contribution graph) → +50 coins if first-
time link (idempotent); Connect LinkedIn (pulls job title + education) → +50 coins if first-time 
link 
3. BROWSE MARKETPLACE: Full-text search via PostgreSQL tsvector GIN index → filter by skill 
category, level (BEGINNER/INTERMEDIATE/EXPERT), price range, 'Available Now' (Redis online 
status check) 
4. BOOK A SESSION: Select mentor skill + time slot → system validates student balance ≥ skill price 
→ LedgerService.lockEscrow() → Escrow record created (LOCKED) → Session record created 
(PENDING) → Mentor notified via WebSocket event 'session.booked'

1.3 Async Bounty Flow 
10. User posts bounty: POST /bounties {title, description, rewardCoins} → validate balance ≥ 
rewardCoins → lock reward in escrow → bounty status = OPEN → expires in 7 days 
11. Expert sees bounty on Bounty Board → uploads <2 min video answer via presigned URL (direct 
to Cloudinary/S3, never through NestJS) → POST /bounties/:id/answers {videoUrl} 
12. Asker reviews video answers → POST /bounties/:id/answers/:answerId/accept → LedgerService 
releases escrow to responder (minus 5% tax) → bounty status = CLOSED 
13. Unanswered bounties auto-expire after 7 days via NestJS Scheduler cron job (@Cron 
EVERY_HOUR) → escrow refunded in full to poster → poster notified

1.5 Full Data Flow Architecture

WebSocket connections. The NestJS backend is the single source of truth for business logic. Redis 
handles ephemeral state (online presence, session caching). PostgreSQL via Prisma handles all 
persistent, ACID-critical data. LiveKit Cloud manages WebRTC rooms independently.

== AUTH MODULE == 
AuthService methods: 
  register(dto: RegisterDto): hash password with bcrypt rounds=12, create user (tokenBalance starts 
at 50 per schema default), call ledger.credit(userId, 50, EARN) to create the transaction record, 
return { user, accessToken, refreshToken } 
  login(dto: LoginDto): validate credentials, generateTokens 
  generateTokens(userId, role): accessToken (JWT, 15m, payload: {sub, role}), refreshToken (JWT, 
7d, payload: {sub}) 
  refreshTokens(refreshToken): verify, generate new pair 
  logout(userId): invalidate refresh token (store hash in Redis with TTL 7d) 
  githubCallback(profile): find or create user by github_id, if NEW github link (github_id was null): 
call ledger.credit(userId, 50, EARN) — IDEMPOTENT check: only credit if user.githubId was null 
before this call 
  linkedinCallback(profile): same pattern as github 
 
JWT Strategy: extract Bearer token from Authorization header, validate sub exists in DB, attach full 
user to request 
JwtAuthGuard: extends AuthGuard('jwt') 
RolesGuard: checks request.user.role against @Roles() decorator metadata 
 
GitHub OAuth Strategy (passport-github2): clientID/Secret from env, callbackURL 
/auth/github/callback, scope ['user:email', 'read:user'] 
LinkedIn OAuth Strategy: clientID/Secret from env, callbackURL /auth/linkedin/callback, scope 
['r_emailaddress', 'r_liteprofile']
